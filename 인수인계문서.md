## 이번 세션에 한 작업과 "왜" 했는지

### 1. 계산 버그 2건 수정 (사용자가 직접 발견)
- **베이스라인 주차 버그**: 증권사가 8주(은행 것)를 잘못 쓰고 있었음 → 4주로 수정.
  `records[-8:]` → `records[-4:]` (analyzer.py, agents/securities/analyzer.py)
  사용자 피드백: "증권 계산로그보니까 8주를사용하네? 이거 버그임" — **은행은 8주, 증권/매스는 4주**가 합의된 방법론이므로 절대 섞으면 안 됨.
- **계산 로그 표시 버그**: 로그에 옛날 퍼센트 공식 `(cur÷base-1)×100`이 표시되고 있었는데 실제 계산은 정규화 절대변화 공식을 쓰고 있었음 → 로그를 실제 계산식 `(cur - base) ÷ mabs = value` 형태로 일치시킴.

### 2. KRX 운영 이슈
- `patch_all_weeks` 실행 시 이모지(🔄✅⚠️) 출력이 cp949 콘솔에서 깨짐 → `[OK]`, `[WARNING]`, `[N라운드]` 텍스트 라벨로 교체 (krx_data_fetcher.py)
- 재로그인 주기 55분 → **25분**으로 변경 (세션 만료가 30분이라 여유있게): `_SESSION_LIMIT = 25`
- **IP 차단 발생**: 과도한 재시도로 KRX가 당일 IP 차단함. "익일 자동 해제"가 알려진 패턴 (다음날 사용자가 데이터로 접속 확인하며 차단 사실 확정함). 빵꾸난 5개 주차(1.5-1.9, 1.19-1.23, 1.26-1.30, 2.2-2.6, 2.9-2.13) 패치는 차단 풀린 후 재시도 필요.

### 3. ETF 매핑 시스템 전면 개편 — 가장 큰 작업
**문제**: 갑자기 매핑이 안 되기 시작함 → 원인은 KRX 캐시 코드에 `*001` 접미사가 붙어서 (`069500*001`) 비교 시 매칭 실패. `.split("*")[0].strip()` 정규화로 해결.

**더 큰 문제 인식**: 그동안 매번 실시간으로 키워드 매칭해서 비교군을 찾고 있었는데, 이는 (a) 부정확하고 (b) 매번 반복 계산이라 비효율. 사용자 지시: "애초에 종목들 비교해서 매핑시켜논다음에 거기서 꺼내오는식으로... 3개 채널 모두가 공유해서 쓸수있게".

**검증 방법론 (사용자가 직접 교정함)**: 처음엔 순매수(투자자별 자금흐름) 데이터로 상관계수를 계산했는데 결과가 안 좋았음(34%만 ≥0.7). 사용자 피드백: "투자자별 컬럼은 왜신경씀? 매핑이랑 컬럼은별개잖아? 단순히 ETF가 유사한지는 가격 상관계수로 판단하면 되잖아" → **ETF 가격 수익률(price return) 상관계수**로 전환 (FinanceDataReader 사용) → 88%가 ≥0.7, 평균 0.896으로 대폭 개선.

**임계값**: 사용자가 0.5 거부하고 **0.7로 명시적 확정**: "응 0.7로하자 0.5는 진짜아닌거같음"

**결과물**:
- `etf_mapping_loader.py` — 3채널(증권/은행/매스) 공용 매핑 조회 모듈. 모듈 레벨 캐시, `get_competitors(code)`, `*001` 정규화 내장
- `etf_mapping.json` — 236개 KODEX ETF 중 147개에 ≥0.7 상관계수 검증된 경쟁 ETF 매핑 (89개는 정말로 유사 경쟁상품 없음 — "코덱스만의 기상천외 전략" 다수 존재)
- 신규 상장 삼성전자/SK하이닉스 단일종목 레버리지 ETF 2개 추가 매핑 (가격 이력 부족으로 corr=null)
- 구버전 `bank_etf_mapping.json` (76개, 은행 폴더 내 하드코딩) 삭제 — 통일했으니 구버전 버림 ("응 통일했으면 구버전버려")

### 4. 공용 계산 모듈 `did_calculator.py` 생성 — 가장 중요한 구조 변경
사용자 통찰: "did계산은 공용으로 하면 안되는거 아님? 증권/개인은 같이해도 은행은 계산방법론이 다르잖아 8주이기도 하고"

→ 증권사와 매스(개인)는 **컬럼명만 다르고 계산 공식이 완전히 동일**(4주 베이스라인 + 정규화 절대변화 DiD)하다는 점에 착안해 `MarketingAnalyzerBase` 추상 클래스로 추출.
- `TARGET_COLUMN`, `BASELINE_WEEKS`, `USE_LP_DETECTION`, `CHANNEL_TYPE` 만 서브클래스에서 설정
- 증권사: `TARGET_COLUMN="financial"`, `USE_LP_DETECTION=True` (LP 노이즈 있음)
- 매스: `TARGET_COLUMN="individual"`, `USE_LP_DETECTION=False` (개인 컬럼은 구조적으로 LP 노이즈 없음)
- 은행은 8주+2단계 Z-score라 별도 관리 — 공용화 대상 아님 (docstring에 명시)

### 5. 매스(대고객 디지털) Agent 최초 구현
이전 세션까지 빈 폴더였던 `agents/mass/`를 `did_calculator.MarketingAnalyzerBase` 상속 구조로 완성.
- 기준 컬럼: 개인, 베이스라인 4주, LP 감지 없음
- `etf_mapping_loader`에서 공용 매핑 가져옴
- `did_history.save_results(..., channel_type="mass")`로 결과 누적 저장
- `check_base_effect(..., channel_type="mass")`로 기저효과 경고 연동

사용자가 미리 주문한 이유: "나중에 또 개인만들면 너혼자삽질할수도있으니 걍 미리만들어놓으셈" — 실제로 이번 세션에 구현하면서 공용 모듈 덕에 매우 빠르게 완성됨 (의도 적중).

### 6. 누적 결과 DB `did_history.py` 신설 + 채널별 분리
사용자: "분석 결과 누적 저장 DB 이건 핵심이라 무조건 해야지" → 이후 "은행만 방법론이 다르니 db 나눠서 해야하는거 아님?" 추가 지적.

- 증권/매스: `did_history.parquet`, `metric="did"`
- 은행: `bank_zscore_history.parquet`, `metric="zscore"` (Z-score는 DiD와 단위/스케일이 달라 같은 파일에 섞으면 비교가 무의미)
- `_get_path(channel_type)` 라우팅 헬퍼로 자동 분기
- `save_results()`, `load_history()`, `check_base_effect()`, `get_summary()` 모두 channel_type 인자로 동작

### 7. 기저효과(base-effect) 착시 경고 구현
"직전 N주 평균 DiD가 양수였는데 이번 주 갑자기 급락하면" → 이전 마케팅 효과가 베이스라인에 흡수되어 마치 효과가 사라진 것처럼 보이는 착시를 경고.
`did_history.check_base_effect()`: 직전 4주 평균이 +0.3 이상이고 이번 주가 평균-표준편차 이상 급락 시 경고 메시지 반환. 증권사·매스 analyzer의 `_analyze_one()`에 연동됨.

- 케이스5 (계절성): "삼성증권 ETF 설날/추석/가정의달"

## 방법론 핵심 — 다음 세션이 절대 헷갈리면 안 되는 것

| 항목 | 증권사 | 매스(개인) | 은행 |
|---|---|---|---|
| 기준 컬럼 | 금융투자 | 개인 | 은행 |
| 베이스라인 | **4주** | **4주** | **8주** |
| 계산 모듈 | `did_calculator.py` 공용 | `did_calculator.py` 공용 | `agents/bank/analyzer.py` 자체 |
| LP 노이즈 감지 | O | X (구조적으로 없음) | O |
| 최종 지표 | DiD 값 (정규화 절대변화, %p로 표시) | DiD 값 | **Z-score** (DiD를 1단계로 구한 뒤, 과거 DiD 분포로 2단계 표준화) |
| 라플라스 스무딩 (1단계 분모) | `평균(\|값\|) + 10억원` | 동일 | 동일 (`+ 1_000_000_000`, `max()` 아닌 `+`로 항상 적용) |
| 2단계 분모 스무딩 | 해당없음 | 해당없음 | `σ + ALPHA_STD(0.01)` — 1단계와 다른 스케일이라 별도 상수 사용 (DiD값은 작은 비율이라 10억을 쓰면 무의미) |
| 결과 DB | `did_history.parquet` | `did_history.parquet` | `bank_zscore_history.parquet` (분리) |

**공식**: `(현재값 - N주평균) ÷ (N주 평균절댓값 + 10억)`
**라플라스 스무딩은 반드시 `+`로 적용** — `max(평균절댓값, 10억)`이 아님. 작은 값일 때도 분모에 10억이 항상 더해져야 함.

### 마케팅 채널 수집기 (collector.py) — 채널별 수집 가능/불가 확정 현황

| 채널 | 상태 | 이유 |
|---|---|---|
| Instagram | ❌ success=False / LOGIN_REQUIRED | 비로그인 차단 구조적 — Graph API 필요 |
| 카카오채널 | ❌ success=False / SPA_STRUCTURE | /posts SPA, JS 렌더링 없이 불가 |
| 삼성증권 이벤트 | ❌ success=False / BOT_DETECTED | SPA 봇 차단, Selenium도 막힘 |
| 구글 트렌드 | ❌ success=False / BOT_DETECTED | pytrends 400/429 봇 차단 |
| 퇴직연금 PDF | ❌ success=False / LOGIN_REQUIRED | 로그인 필요 |
| 네이버 블로그 | ✅ 모바일 RSS 폴백 있음 | 403 시 m.blog.naver.com으로 재시도 |
| TIGER/SOL YouTube | ✅ 검색 폴백 있음 | 핸들 추출 실패 시 검색어로 채널 ID 찾음 |

**절대 하지 말 것**: 수집 실패 시 뉴스 검색/연관 키워드 등 완전히 다른 데이터를 success=True로 반환하는 가짜 폴백. "안되면 진짜 안된다고" — 사용자 명시적 지시. 실패는 success=False로 정직하게.

### 은행 Step3 DiD 성능 (app_bank.py) — 중요

**문제**: 브라우저 새로고침 시 session_state 날아가면 전체 KODEX ETF (100개+) 재계산 → 매번 느림. parquet에는 숫자만 있어서 ETFDiDResult 객체 복원 불가.

**해결**: `.did_cache/bank_{week}.pkl`에 `summary` dict (ETFDiDResult 객체 포함) 전체를 pickle로 저장.
- 첫 실행: 계산 후 pickle 저장
- 이후 리로드: pickle 읽기만 → 즉시 복원
- 주차 변경: 새 pkl 파일 자동 생성

**중요**: 새 세션이 "top-N개만 분석하자"는 식으로 ETF 수를 제한하면 안 됨. 전체 KODEX ETF 전수 분석이 요구사항. 대신 pickle 캐시로 초기 1회만 느리게 해결.

## 사용자(김종우)와 협업 시 유의사항

- **방법론은 채널별로 토론을 거쳐 확정한 것** — 임의로 "통일하면 좋겠다"고 섞지 말 것. 특히 베이스라인 주차(4주 vs 8주)나 최종 지표(DiD vs Z-score)를 혼동하면 강하게 교정받음.
- **계산 검증 시 사용 데이터 종류를 명확히 구분**: ETF 유사도 판단 = 가격 수익률 상관계수, 투자자별 순매수 컬럼과는 무관 (사용자가 직접 교정한 핵심 포인트)
- **공용화 가능한 로직은 미리 만들어 3채널이 공유**하도록 — "너 혼자 삽질하지 말고 미리 만들어놔라"는 명시적 지시 있었음. 이 원칙 덕분에 매스 Agent를 빠르게 완성함.
- 라플라스 스무딩은 `+`로 항상 적용 (`max()` 금지) — 과거에 한번 잘못 구현해서 교정받은 이력 있음
